# Business Logic and Rules

This document extracts booking and availability rules from the implementation (frontend + database + edge functions). Items that cannot be proven from code are labeled.

## 1) Tenant and tier rules
### Location limits by tier
The dashboard enforces a maximum number of locations based on `organizations.tier` (as implemented in `src/pages/dashboard/Locations.tsx`):
- `tier_1`: 1 location
- `tier_2`: 10 locations
- `tier_3`: 100 locations

**Note**: This is enforced in the UI; confirm if backend also enforces.

## 2) Booking slot generation rules (public booking widget)
Primary implementation lives in `src/pages/BookingPage.tsx`.

### Inputs to availability
Slots are computed from:
- **Opening hours**: `location_availability`
- **Off days**: `organization_off_days` (org-wide or location-specific)
- **Public holidays**: derived from `organizations.holiday_region` with overrides via `organization_holiday_overrides`
- **Closures**: `location_closure_slots` (org-wide or location-specific)
- **Break times**: `organization_break_slots` (+ staff mapping) (exact semantics based on flags like `applies_whole_salon`)
- **Busy intervals**: existing bookings + pending holds from RPC `get_location_busy_intervals`
- **Optional Google conflicts**: if Google Calendar is connected and/or staff is selected

### Buffer between bookings
The widget applies a built-in buffer:
- **Buffer = 15 minutes** (`BOOKING_SLOT_BUFFER_MINUTES = 15`)

### Slot stepping
Slots are generated by iterating through available windows in increments of:
- \(serviceDuration + buffer\)

### Multi-service duration and pricing
The widget supports selecting multiple services and computes:
- Total duration = sum of durations
- Total price = sum of prices

**Needs validation**: how multi-service selection is persisted in bookings (single booking vs multiple bookings).

### Staff-related behavior (slot eligibility)
If the customer does not pick a specific staff member, the UI attempts to allow a slot if:
- there exists at least one active staff member at the location not blocked by staff-specific breaks during that window

**Needs validation**: whether server-side booking creation assigns staff or leaves unassigned.

## 3) Conflict and overlap rules (server-side)
### Busy interval computation (source of truth for conflict checks)
Server-side booking creation uses RPC:
- `get_location_busy_intervals(location, start, end, exclude_token)`

This function combines:
- Confirmed bookings
- Pending confirmation holds (active, unreleased)

### Staff overlap rule (when staff is assigned)
The schema includes a constraint to prevent double booking the same staff member:
- A partial unique index on `(staff_id, start_time)` when `staff_id IS NOT NULL` and status is not cancelled (migration `20260311120000_bookings_staff_id_nullable.sql`).

## 4) Booking lifecycle rules
Booking status enum includes:
- `pending`, `confirmed`, `paid`, `cancelled`, `completed`, `no_show`

Observed UI behavior:
- Rescheduling sets status to `confirmed` (Dashboard bookings UI).
- Removing staff from the system can unassign staff from bookings (sets `bookings.staff_id = null`).

## 5) Cancellation/reschedule rules
The dashboard allows:
- rescheduling by choosing a new date/time
- setting status to cancelled/no_show/completed

**Needs validation**
- Whether customers can self-cancel/reschedule via public links (not clearly visible in route list beyond payment cancel flow).
- Any cutoffs (e.g., cannot cancel within X hours) are not observed in the high-level scan.

## 6) Reminder rules
### Defaults and overrides
- Organization-level defaults exist on `organizations` (day-before and hour-before flags).
- Per-customer overrides exist in `customer_reminder_preferences`.

### Dedupe
Reminders use `booking_reminder_sent` to prevent duplicates.

### Scheduling
Reminders are sent by edge function `send-booking-reminders` which expects an external scheduler calling it with `x-cron-secret` matching `CRON_SECRET`.

## 7) Holidays and regional calendars
- Each organization can set a holiday region (`organizations.holiday_region`).
- Public holidays are computed based on that region and can be overridden via `organization_holiday_overrides`.

## 8) Google Calendar rules (high-level)
- If connected, the dashboard calendar can fetch and show Google Calendar events.
- Booking entries can be synced to Google Calendar via edge functions.

**Needs validation**
- Whether GCal events are treated as “blocking” availability universally or only in staff-selected flows.
