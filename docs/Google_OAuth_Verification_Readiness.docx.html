2. What to confirm or fix before submitting

2.1 Privacy Policy content and URL

  • URL: The policy must be on your domain (e.g. https://salonora.eu/privacy or https://salonora.eu/privacy-policy).
  • Same URL everywhere: The exact same URL must be linked from the homepage and set as “Privacy Policy” in the OAuth consent screen in Google Cloud Console.
  • Google data: The policy must explicitly say that you use Google Calendar to read calendar events (to block time slots) and create events (for confirmed bookings), and that this data is not used for ads, not sold, and only for providing the booking/calendar feature.

If the current policy does not mention Google/Calendar, add a short “Google Calendar” or “Third-party services” section before you submit.

2.2 Domains in Google Cloud

  • Authorized domains (OAuth consent screen): Add both salonora.eu and booking.salonora.eu (no http/https, just the domain).
  • Redirect URIs (OAuth client): Add every URL where Google redirects after the user approves. If the callback is on Supabase (e.g. https://pgcvqaexvnwwskdhooly.supabase.co/functions/v1/google-auth-callback), that exact URI must be in the OAuth client’s “Authorized redirect URIs”. If you use a custom redirect on your domain, that URI must be there instead (or as well).

2.3 Use HTTPS for the booking app

You mentioned http://booking.salonora.eu. For production and verification, the app should be served over https://booking.salonora.eu. In the consent screen, use https for any app/homepage URLs.

2.4 Domain ownership (Search Console)

salonora.eu (and ideally booking.salonora.eu if it’s a separate property) should be verified in Google Search Console. The Google account that verifies the domain should be owner or editor of the same GCP project you submit for verification. Until this is done, verification can be rejected for “domain ownership”.

2.5 OAuth consent screen and app info

  • Application name: e.g. “Salonora” (consistent with the site).
  • Logo: 120×120 px, matches the brand.
  • User support email: Real, monitored (e.g. info@salonora.nl from the site).
  • Developer contact: Email(s) Google can use for verification issues.
  • Scopes: Only Calendar + Calendar events, with a short justification (e.g. “Sync bookings and block availability with the user’s Google Calendar”).

2.6 Demo video (sensitive scope)

Show the full flow: user on booking.salonora.eu (or wherever they connect Google) → “Connect Google Calendar” → full OAuth consent screen in English (same scopes you request) → back to your app → calendar view / blocked slots / a booking synced to Google Calendar. Upload (e.g. unlisted YouTube) and use the link in the verification form.

3. Summary

Homepage and brand: salonora.eu and publishing the app on booking.salonora.eu are fine for verification as long as the checklist above is completed.

Before you “put verification on” (submit for verification): Complete the Privacy Policy (same URL + Google Calendar disclosure), authorized domains and redirect URIs, HTTPS for booking.salonora.eu, domain ownership in Search Console, consent screen details, and the demo video. Then submit.
